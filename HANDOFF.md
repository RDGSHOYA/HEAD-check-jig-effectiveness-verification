# HANDOFF

## 1. アプリ概要

このアプリは、HEADチェック導入によってHEAD交換が不要になった件数を入力すると、HEAD種類ごとに交換HEAD金額、従来作業金額、新作業金額、削減HEAD費用、削減工数金額、工数効果額、正味効果額、削減率を即時計算して表示する静的ダッシュボードです。年間約5億円規模のHEAD交換コストに対して、シリンジチェックやHEADチェックモニターによる交換回数最適化の効果を、経営層向けに見やすく把握することを目的にしており、入力内容はブラウザ内に保存されて次回起動時に復元されます。

## 2. ローカルでの実行方法

このプロジェクトは依存関係なしの静的HTML/CSS/JavaScriptです。この環境でローカル確認する場合は、PowerShellで以下を実行してブラウザで `http://localhost:8000/index.html` を開いてください。

```powershell
cd C:\Users\satsumi\Documents\Codex\2026-06-08\ai-6-1-2-3-4\outputs
C:\Users\satsumi\.cache\codex-runtimes\codex-primary-runtime\dependencies\python\python.exe -m http.server 8000
```

通常のPythonがPATHに入っているPCでは、2行目を `python -m http.server 8000` に置き換えても構いません。サーバーを使わずに `index.html` をブラウザで直接開いても基本動作します。

## 3. ファイル構造

- `index.html`: 画面のHTML構造です。KPIエリア、グラフエリア、明細テーブル、合計行の土台を定義しています。
- `styles.css`: 画面全体の見た目を定義しています。ウォームライトグレー背景、白カード、KPI、グラフ、横スクロールなしの明細テーブル、レスポンシブ表示、削減HEAD費用・削減工数金額の青色表示を担当します。
- `app.js`: HEADマスタ、入力状態、LocalStorageへの保存・復元、計算ロジック、KPI・グラフ・明細テーブル・合計行の描画処理を担当します。
- `README.md`: 簡易的な起動説明です。
- `BRIEF.md`: 元の仕様メモです。文字化けして見える場合がありますが、UTF-8で読むと日本語仕様として確認できます。
- `HANDOFF.md`: この引き継ぎ文書です。

## 4. 未完成・ダミー・プレースホルダ

- HEADマスタは `app.js` 内に固定データとして直書きされています。外部CSV、JSON、API、データベースからの読み込みには未対応です。
- 入力データはReactやAPIではなく、素のJavaScriptとブラウザのLocalStorageで管理しています。同じブラウザで開くと前回入力値が復元されます。
- グラフはライブラリを使わず、HTML/CSS/SVGで簡易描画しています。Power BI相当の高機能なチャートではありません。
- 初期値はすべて0件です。実績データやサンプルデータは入っていません。
- GitHub PagesやNetlifyなどへの公開設定はまだありません。

## 5. 既知のバグや制約

- 横スクロールをなくすため、明細テーブルは小さめの文字と折り返し表示にしています。画面幅が狭い場合は読みづらくなる可能性があります。
- 列幅調整機能は一度追加しましたが、横スライドをなくす要望に合わせて外しています。
- `交換不要件数` は `交換件数` を超えないよう自動で丸めますが、エラーメッセージ表示はありません。
- 保存先はブラウザのLocalStorageのみです。別ブラウザ、別PC、シークレットモード、LocalStorage削除後は復元できません。
- 金額や率は画面表示用に丸めています。内部計算は整数ベースですが、削減率は小数1桁表示です。
- 新作業金額は仕様どおり `（120分 + 36分）× 交換件数 × 工数単価` として計算しています。交換不要時の76分との差分は `削減工数金額` として別列に出しています。

## 6. Webチームへの推奨次ステップ

1. 計算式の業務定義を関係者と再確認する。特に「新作業金額」「削減工数金額」「工数効果額」「正味効果額」の関係は、報告資料で誤解されやすいので用語を固定する。
2. テーブル表示を改善する。列数が多いため、PC向けには固定ヘッダー、列表示切替、重要列の強調、または詳細行の折りたたみを検討する。
3. 入力値の保存方法を拡張する。現在はLocalStorageのみなので、複数人利用や履歴管理が必要であればCSV入出力、バックエンド/API連携、またはクラウド保存を検討する。
4. グラフを本格化する。Recharts、Chart.js、EChartsなどを導入すると、ツールチップ、凡例、軸、レスポンシブ対応が安定する。
5. 静的ホスティングを設定する。GitHub Pages、Netlify、または社内Webサーバーに置けるように公開手順を整備する。
6. テストを追加する。計算ロジックを関数として分離し、代表ケースの単体テストを作ると、仕様変更時の事故を減らせます。
